1. The agent more or less fully implemented the spec. There were some privacy and UX considerations that weren't fully addressed and were caught by the reviewer, but these were honestly not talked about in the spec.
2. During phase 2, I didn't need to intervene.
3. The review didn't really give feedback on style (I would have liked to see less dicts and more OOP) and missed some glaring UX things. For example, the questions the llm made were short-answer questions, and were only considered correct if the user entered the exact expected answer. This is obviously not reasonable. That said, the reviewer caught some good privacy considerations and error handling stuff.
4. I would have talked more about style, in particular using OOP. I also would have been extra descriptive/thoughtful about the user flow of the app and the question design, as the llm was NOT good at inferring that.
5. I think plan-delegate-review has advantages and disadvantages. It is painful and time consuming to type out a spec, and honestly I think certain details can hurt or provide negligable benefits. For example, if I specify a certain file structure but during the implementation, it turns out it makes sense to add some more files, or move things around, I think the llm would be hesitant to do that because it violates the spec. But for UX things that it got wrong, I think the spec would have been super helpful if I had really honed in on that and been really specific. In my opinion, implementation details (especially relatively standard/trivial things) can be left somewhat underspecified, but specs might be quite useful for UX and code style things that llm tends to overlook.
